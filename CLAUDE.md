# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Koru** (`com.dev.koru`) is an Android-only minimalist launcher + mindful app blocker.
It is a **dual-runtime app**: a Flutter/Dart UI layer and a substantial native
Kotlin/Android layer that does the actual enforcement. Neither side is optional —
the blocking only works because of the Kotlin services, and the UI/config only
exists in Dart. Most architectural bugs live at the seam between the two.

## Commands

Run from the project root (`new_app`). Shell is PowerShell.

```sh
# Run on a connected device (the only target — no iOS/web/desktop)
flutter run

# Static analysis (lints from flutter_lints via analysis_options.yaml)
flutter analyze

# Dart/Flutter tests — whole suite
flutter test

# A single test file / single test by name
flutter test test/path/to/foo_test.dart
flutter test test/path/to/foo_test.dart --plain-name "describes the case"
```

### Installing / updating on the device (REQUIRED check)

Before installing the APK, **always check whether `com.dev.koru` is already on the
device, and if so update it in place instead of doing a fresh install** — a plain
`adb install` of an already-present app fails with `INSTALL_FAILED_ALREADY_EXISTS`,
and a clean reinstall wipes the Drift DB / Hive settings and forces re-granting the
AccessibilityService + overlay + device-admin permissions every time. `adb install -r`
replaces the app **keeping its data and granted permissions**.

```powershell
# PowerShell — build, then update-in-place if installed, else fresh install
flutter build apk --debug
$apk = "build/app/outputs/flutter-apk/app-debug.apk"
if (adb shell pm list packages | Select-String "com.dev.koru") {
    adb install -r $apk     # -r = reinstall, preserves data + permissions
} else {
    adb install $apk
}
```

`flutter run` and `flutter install` already do the right thing (incremental update,
no data wipe) — prefer them for normal dev. The explicit check above is for when you
drive `adb install` directly (e.g. installing a prebuilt APK).

### Code generation (REQUIRED after touching generated sources)

Drift tables/DAOs, Freezed models, Riverpod `@riverpod` providers, and Hive
adapters are all code-generated (`*.g.dart`, `*.freezed.dart`). After editing any
table, DAO, `@freezed` class, `@riverpod` provider, or Hive type, regenerate:

```sh
dart run build_runner build --delete-conflicting-outputs
```

Localization (`lib/l10n/*.arb` → `lib/l10n/generated/`) is generated by
`flutter gen-l10n` (config in `l10n.yaml`); `flutter run`/`build` also trigger it.

### Kotlin / Android unit tests

Use the **module-scoped** task and skip the Flutter compile step (full build is slow
and unnecessary for pure-Kotlin tests). Run from `android/`:

```sh
./gradlew :app:testDebugUnitTest -x compileFlutterBuildDebug
```

Results XML lands in `build/app/test-results/`. Robolectric has no Android Keystore,
so tests touching `EncryptedSharedPreferences` skip or hit a fallback. If Gradle
complains about "requires experimental feature", run `flutter clean` first.

## Architecture

### Layering (Dart, under `lib/`)

Clean-ish layering, dependencies point inward:

- `core/` — cross-cutting: `di/providers.dart` (root Riverpod providers + overrides),
  `router/app_router.dart` (go_router + `KoruRoutes`), `theme/`, `constants/`,
  `diagnostics/` (the BlackBox logger).
- `data/` — `database/` (Drift: `app_database.dart`, `tables/`, `daos/`), `local/`
  (Hive key-value settings), `repositories/`, `models/`.
- `domain/` — pure entities + usecases, no Flutter imports.
- `presentation/` — `screens/`, `widgets/`, and `providers/` (Riverpod state).
- `platform/` — the Dart side of the platform channels (see below).

State management is **Riverpod**; navigation is **go_router**. `lib/app.dart`
(`KoruApp`) is the root `ConsumerWidget` and watches several "always-on" providers
(lifecycle invalidator, blocking/package event refreshers, home-intent listener,
achievement evaluator) — these are side-effecting roots, not UI.

### The two storage systems

- **Drift / SQLite** (`koru.db`) — relational, owns all profile/blocking/stats data.
- **Hive** (`HiveSettingsService`) — flat key-value app settings (theme, font, flags).

Both are constructed in `main.dart` and injected into `ProviderScope` overrides.

### Native enforcement (Kotlin, under `android/app/src/main/kotlin/com/dev/koru/`)

The blocking engine runs **outside the Flutter process**:

- `service/KoruAccessibilityService.kt` — primary engine. Event-driven on
  `TYPE_WINDOW_STATE_CHANGED`; when a blocked app surfaces under an active profile it
  shows a native Compose overlay (`service/OverlayManager` + `service/OverlayView`)
  and performs `GLOBAL_ACTION_HOME`. Reads profile state via an atomic
  `ProfilesSnapshot` published with `AtomicReference`.
- `service/LockForegroundService.kt` — **backup** polling loop in the main process,
  so enforcement survives OEMs (ColorOS/MIUI/Samsung) killing the AccessibilityService.
- `channels/` — Flutter MethodChannel/EventChannel handlers. `BlockingMethodChannel`
  delegates to focused handlers under `channels/blocking/`.
- `strictmode/` — DeviceAdmin-backed strict mode, backdoor codes, unblock tokens.
- `content/` + `browser/` — in-app section detection (Instagram/YouTube) and browser
  URL extraction for website blocking.
- `KoruEngineManager.kt` — caches a single `FlutterEngine` so recreating
  `MainActivity` does **not** re-run Dart `main()` (cold-start avoidance). See
  `MainActivity.onCreate`.

### CRITICAL: the cross-runtime DB contract

The same `koru.db` file is opened by **two SQLite runtimes**: Drift (Dart) and raw
Kotlin (`android/.../db/NativeDatabase.kt`). Read the long header comment in
`lib/data/database/app_database.dart` before touching the schema. Non-negotiables:

1. **Drift owns the schema** — it is the only side that runs CREATE/ALTER. Kotlin
   reads with hardcoded column names and **fails silently** (empty result →
   enforcement stops, no error) if you rename/drop a column it depends on.
2. Renaming a "native" column requires **three coordinated edits**: the Drift table
   (+ migration), `android/.../db/DbSchema.kt`, and the `kNativeSchemaContract` map in
   `test/data/database/db_schema_contract_test.dart`. That test opens the live Drift
   schema and asserts every (table → columns) the native side needs still exists —
   it is your guardrail; keep it green.
3. **`journal_mode = DELETE`, never WAL** — the two SQLite libs don't share `-wal`/
   `-shm` files reliably; WAL causes `SQLITE_IOERR_SHM*`. `busy_timeout = 5000` on
   both sides.

### Platform channels

`lib/platform/platform_channel_service.dart` (`PlatformChannelService`) is a façade
grouping all channels (`blocking`, `profile`, `permission`, `strictMode`, `events`),
exposed via `platformChannelServiceProvider`. Add new native calls inside the matching
channel class rather than scattering `MethodChannel` instances.

### Launcher vs. shell routing

The launcher UI (clock + favorites + drawer, `/launcher`) lives **outside** the
bottom-nav shell and is only shown when Koru is invoked via the HOME intent
(`MainActivity` sets the initial route accordingly). The dashboard tabs
(`/home`, `/profiles`, `/focus`, `/stats`, `/settings`) live inside the shell.
System-gesture overrides for the launcher are scoped to its route via `RouteAware` —
never enabled at widget mount. Route names are centralized in `KoruRoutes`.

## Conventions

- The whole `flutter test` suite **and** the Kotlin suite must be green before any
  push to `main` — including pre-existing failures.
- Keep the three enforcement paths (app blocking / website blocking / in-app content)
  aligned: their guards (`isProfileActiveNow`, `isBypassed`, etc.) drift apart easily.
- `docs/` holds point-in-time review and performance-analysis reports; they are
  historical, not living specs.
