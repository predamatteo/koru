# Koru — Security / Bug Bounty / Code / Architecture Review

**Date:** 2026-05-26
**Scope:** Full app — Flutter/Dart (`lib/`, ~34k LOC) + native Android Kotlin (`android/app/src/main/kotlin/com/dev/koru/`, ~8k LOC).
**Method:** 3 specialized agents in parallel, read-only — Security Engineer (security + bug bounty), Software Architect (architecture), Code Reviewer (correctness). This document is the orchestrator's consolidated, deduplicated, cross-referenced synthesis.

**Threat model note:** Koru is a *self-control / commitment* app. The primary adversary is frequently **the user themselves** trying to defeat their own limits. Therefore **bypass resistance is a security property**, not just UX. A secondary adversary is a malicious co-installed app.

---

## Executive summary — two structural root causes

Three independent reviews converged on **two** structural problems that together account for nearly every Critical/High finding:

### Root cause A — No single source of truth for "is X blocked right now"
The blocking decision is re-implemented across **4 code paths** (app / website / in-app-content in the AccessibilityService, plus the polling backup in `LockRunnable`) with **no shared evaluator**. `isProfileActiveNow` alone exists in **three** divergent copies with different interval semantics. This is the *architectural* root cause (**ARCH-01**) behind the recurring "parity" bug noted in project memory, and it directly produces the correctness Criticals: the backup path silently enforces nothing for focus/website/section blocking (**CR-01**, **ARCH-02**), interval boundaries differ by path (**CR-02**), the backup ignores the Wi-Fi constraint (**CR-03**), and the in-app-content path ignores active bypasses (**CR-07**). It also amplifies **SEC-03** (the cap logic isn't centralized or monotonic-guarded).

### Root cause B — Authorization & integrity live in the wrong layer / rely on the wall clock
The strict-mode disable gate (backdoor-code auth) is enforced **only in the Dart UI**, while the native channel accepts `setStrictModeOptions(0)` unconditionally (**SEC-01** — full strict-mode bypass). Separately, the daily hard cap, the brute-force lockout, and the bypass-friction counter are all anchored to `System.currentTimeMillis()`, so a backward clock change resets them (**SEC-03/04/05**) — the exact attack the `BypassStore` was hardened against, but which the cap logic doesn't inherit. And clearing app data wipes all enforcement state (**SEC-02**).

## Critical findings

### SEC-01 — Strict-mode disable gate enforced only in Flutter UI; native `setStrictModeOptions` accepts mask=0 unauthenticated
Full strict-mode bypass with no code/lockout/device-admin step. `StrictModeMethodChannel.kt:100` clears the mask unconditionally; the only auth is Dart-side at `lib/presentation/screens/settings/sub_screens/strict_mode_screen.dart:201-208`. Contrast `disableDeviceAdmin`, which *does* enforce the guard natively (`StrictModeMethodChannel.kt:80-89`).
**Fix:** Move authorization into the native channel — reject any mask change that *clears* an active bit unless accompanied by a freshly-validated, single-use backdoor token (native-issued, `elapsedRealtime`-bound, short TTL, consumed atomically). Adding restrictions stays unauthenticated.

### CR-01 / ARCH-02 — Backup enforcement path (`LockRunnable`) is an asymmetric subset: no focus/quick-block, no website, no in-app-section
`LockRunnable.checkAndBlock` (`LockRunnable.kt:139-267`) enforces only app-block + daily-limit. It never reads `QuickBlockStore` (the only reader is `KoruAccessibilityService.kt:848`) and has no website/section logic. So in the exact scenario the backup exists for — OEM kills the AccessibilityService — **Pomodoro/focus, website blocking, and Reels/Shorts blocking silently stop** while the UI says the profile is enforced.
**Fix:** Wire focus/quick-block (and, where feasible without the a11y node tree, the rest) into the backup; or make the asymmetry an explicit, user-surfaced design decision (banner: "website/section blocking paused while service is down"). Best done via the shared evaluator below.

### CR-02 / ARCH-01 — `isProfileActiveNow` implemented 3× with divergent interval semantics; no shared policy evaluator (ROOT CAUSE)
Three copies: `KoruAccessibilityService.kt:1240-1247` (half-open `[from,to)`, `from==to` ⇒ 24h), `LockRunnable.kt:308-312` (closed `[from,to]`, `from==to` ⇒ 1-min), `schedule_utils.dart:19-22` (half-open, `from==to` ⇒ never). At the boundary minute and the `from==to` case, the three disagree — behavior depends on which engine is alive.
**Fix:** Extract one pure `BlockPolicyEvaluator` (no Android deps) returning a sealed `BlockDecision`; route all 4 native call sites + the Dart "active now" check through equivalent shared logic. Add a parity test feeding identical `(profile, interval, now)` to both Kotlin paths asserting equality at `from-1/from/to-1/to/midnight/from==to`. **This single refactor structurally eliminates the entire parity bug class** (CR-01/02/03/07).

## High findings (condensed — full detail in agent reports)

- **SEC-03 — Daily hard cap is wall-clock based; clock rewind unblocks the strict cap.** `UsageCounter.todayForegroundMs` (`UsageCounter.kt:32-43`) defines "today" from `Calendar.getInstance()` with no monotonic cross-check; the "absolute" cap at `KoruAccessibilityService.kt:905-908` reads ~0 after a backward date change. *Fix:* persist `(lastWallMs, lastElapsedMs, accumulatedTodayMs)` like `BypassCountStore`; on backward wall jump don't roll the day over; add date/time settings to the strict block list. **(Near-Critical: defeats the strongest time-based commitment.)**
- **SEC-02 — "Clear app data" wipes all bypass-resistance state** (bypasses, caps, encrypted strict mask, used-codes, lockout). Inherent to Android without Device Owner, but mitigatable: on resume/start, fail toward "blocked" when a durable "was-enabled" signal exists but local state is empty; detect the wipe via a Device-Admin-held marker. (`allowBackup=false` already blocks `adb backup` — good.)
- **SEC-04 — Backdoor brute-force lockout resettable by a clock jump** (`StrictModeMethodChannel.kt:333-345`). The 40-bit keyspace isn't hand-brute-forceable, but the lockout is the safety margin. *Fix:* anchor lockout to a reboot-corrected monotonic baseline; never let a forward wall jump shorten it.
- **CR-03 — Backup path ignores the Wi-Fi/SSID profile constraint** (present at `KoruAccessibilityService.kt:1256-1260`, absent in `LockRunnable`). Wi-Fi-scoped profiles behave oppositely when a11y is dead. *Fix:* fold into the shared evaluator (CR-02).
- **CR-04 — `BypassCountStore` read cache never invalidates + unsafe in-place `JSONObject` mutation** (`BypassCountStore.kt:194-195`, `:85-103`). Main-process UI count is permanently stale; in-place mutation of the shared cached instance is a data race. *Fix:* `(lastModified)` invalidation + copy-before-mutate + adopt `BypassStore`'s file-lock RMW.
- **CR-05 — `NativeDatabase.close()` races active cursors in the main process.** `LockForegroundService.triggerProfileReload` (`:47-53`) closes on the main thread while `LockRunnable` iterates cursors on the BlockingThread → transient empty/partial profile data = brief enforcement gap. *Fix:* don't `close()` to force reload; set `needsReload` and reopen on the worker thread.
- **ARCH-03 — Six cross-process stores, six robustness levels, no shared abstraction.** Only `BypassStore` is atomic + locked + cache-invalidated; the other 5 range down to plain `writeText` with no invalidation. *Fix:* one `FileBackedStore<T>` with BypassStore-grade guarantees; migrate the other 5.
- **ARCH-04 — Two runtimes own one SQLite file with no schema contract.** `NativeDatabase.kt` hardcodes ~40 column names; a routine Drift migration compiles fine and breaks native enforcement silently at runtime. The `journal_mode=DELETE` agreement is an undocumented cross-runtime invariant. *Fix:* shared schema-constants + CI check that native columns match Drift's generated schema; document the journal-mode invariant.

## Medium findings (condensed)

- **SEC-07 — Full browsing URLs logged to logcat at INFO** (`KoruAccessibilityService.kt:1129`, `full=${detected.fullUrl}`), on every throttled content-change. Privacy leak for a wellbeing app. *Fix:* gate behind `BuildConfig.DEBUG`, log domain only, never the full URL/query. **(Cheapest high-value fix.)**
- **CR-07 — In-app-content path missing the bypass guard the website path has** (`checkInAppContentBlocking`, `KoruAccessibilityService.kt:1047-1111` vs website guard at `:1164`). A granted section bypass (Reels/Shorts) is dead-on-arrival — user is re-blocked on the next scroll. *Fix:* mirror the `OverlayManager.isBypassed` guard scoped to the section wireId.
- **CR-06 — `from==to` interval is user-reachable and self-contradicting** across the 3 paths; the `isAllDayAuto` column (`intervals_table.dart:12`) is never read. *Fix:* decide the meaning, wire `isAllDayAuto` through or drop it, validate in the picker.
- **CR-08 — Browser content-change events skip the app/limit re-check** (`KoruAccessibilityService.kt:496-514` calls only `checkWebsiteBlocking`). Mostly mitigated by `scheduleLimitCheck`. *Fix:* run `checkAppBlocking` first in that branch.
- **SEC-05 — Bypass-count friction resets on backward clock / post-reboot forward jump** (`BypassCountStore.kt:155-165`).
- **SEC-06 — Used-codes set evicts arbitrary entries at cap=100** (`StrictModeMethodChannel.kt:266-275`), weakening single-use replay protection. *Fix:* make the DB `used_backdoor_codes` table authoritative.
- **ARCH-05 — God objects:** `KoruAccessibilityService.kt` (1442 LOC) and `OverlayManager.kt` (1011 LOC, mixes Compose UI + WindowManager lifecycle + bypass policy). Nearly untestable as units.
- **ARCH-06 — Hand-synced magic numbers:** strict-mode bits duplicated (`StrictModeEnforcer.kt:26-30` vs `StrictModeStore.kt:53-57`), `SETTINGS_PACKAGES` duplicated, restriction-type ints as raw literals. *Fix:* one shared `BlockingContract`.
- **ARCH-09 — God-facade channels:** `BlockingChannel` (Dart, 292) / `BlockingMethodChannel` (Kotlin, 558, 38 methods) bundle ~7 concerns.
- **ARCH-07 — `domain/` layer dependency leaks:** `evaluate_achievements.dart:1` imports `data/`; entities import `package:flutter/material.dart`. The domain layer points "up."

## Low / Info findings

- **SEC-08** — Homemade `SHA-256(constant‖value)` instead of real keyed HMAC (`StrictModeStore.kt:157-161`); redundant since value is in EncryptedSharedPreferences, but the comment overstates protection (key is in the binary).
- **SEC-09** — Non-atomic write fallback in `BypassStore.writeAtomic` (`:184-190`) and plain `writeText` in `AppUsageLimitsStore.save` (`:112`)/`QuickBlockStore` — torn limits file ⇒ all caps vanish. *Fix:* temp+rename everywhere; fail toward "keep last known caps."
- **SEC-10** — Legacy 32-bit ANDROID_ID-derived backdoor code reachable as Keystore-failure fallback (`BackdoorCodeGenerator.kt:57,127-135`). *Fix:* fail secure on Keystore failure.
- **SEC-11** — `QuickBlock` focus expiry is wall-clock based (`QuickBlockStore.kt:46`); forward jump ends focus early.
- **SEC-12** — `EXTRA_REQUIRE_BACKDOOR_CODE` set by `KoruDeviceAdminReceiver.onDisableRequested` (`:91`) is never read by `MainActivity` → the documented deterrent doesn't fire (not exploitable; system-only intent).
- **CR-09** — State-bearing writes swallow exceptions (`catch (_: Exception) {}`) — a failed `BypassStore.put` reports success to the UI then re-blocks. *Fix:* propagate failure for state-bearing writes.
- **CR-10** — Zero unit tests for any block-decision function (`checkAppBlocking`/`checkWebsiteBlocking`/`checkInAppContentBlocking`/`LockRunnable.checkAndBlock`/both `isProfileActiveNow`). Every Critical/High above is in untested code.
- **ARCH-08** — Four parallel KV mechanisms (Hive, Drift `Settings` table, shared_preferences, native prefs); the Drift `Settings` table has **zero callers** — dead merge debt.
- **ARCH-10** — Stale doc invariants (table count "21" vs 25 registered; "5 stores" vs 6). Where comments encode contracts, make them CI-checkable.

## Verified-solid defenses (checked, found correct — reassurance)

- **`BypassStore` hardening** (all 3 agents): atomic temp+rename, cross-process `FileChannel.lock`, intra-process lock, `(mtime,len)` cache invalidation, dual-clock anti-rollback `isActive`. **This is the template the other stores should follow.**
- **Backdoor code crypto:** `SecureRandom`, 30-symbol alphabet, 8 chars ≈ 40 bits, Keystore-backed EncryptedSharedPreferences (AES256-GCM), validated by lookup with constant-time compare. (Caveat: SEC-10 fallback.)
- **No network / telemetry / PII exfiltration:** exhaustive search for HttpURLConnection/OkHttp/Socket/Firebase/Crashlytics/Sentry → **zero hits** in `main/`. The notification listener only dismisses, never reads content. INTERNET permission is held but unused for exfiltration. (Only logcat exposure, SEC-07.)
- **SQL injection:** all `NativeDatabase.kt` queries use `?` bound params, including dynamic-looking ones. Clean.
- **Exported components:** NotificationListener + DeviceAdmin protected by signature/system bind permissions; `BootReceiver` validates the action; internal broadcasts use `setPackage()`; receivers register `RECEIVER_NOT_EXPORTED`. No injectable privileged path from a co-installed app.
- **Strict-mask fail-secure:** HMAC mismatch on an existing record returns `ALL_OPTIONS_ENABLED` (31), not 0 — correct direction. (Gap only in the empty-store case, SEC-02.)
- **Overlay non-dismissibility:** `TYPE_APPLICATION_OVERLAY`, full-bleed opaque, strict mode shows non-actionable indicator, pause hatch disabled for hard blocks, BACK/HOME/recents re-asserted. Sound — weakness is upstream (the *decision*), not the overlay.

## Recommended remediation roadmap (sequenced)

**Track 1 — Security hardening (ship-blocking for a commitment app):**
1. SEC-01 — native auth gate on `setStrictModeOptions` (one-time token). *Highest severity.*
2. SEC-03 — monotonic guard on the daily-cap day rollover.
3. SEC-04 — reboot-corrected monotonic lockout.
4. SEC-07 — stop logging full URLs (trivial).
5. SEC-02 — fail-toward-blocked on empty-state-after-enabled; SEC-05/06/09 as follow-ups.

**Track 2 — Enforcement correctness (the parity cluster — do the evaluator first):**
1. **ARCH-01 / CR-02** — extract one `BlockPolicyEvaluator` + shared `isProfileActiveNow`; add the parity test. *This is the keystone refactor.*
2. CR-01 / ARCH-02 — wire focus/quick-block (+ website/section where feasible) into the backup path through the evaluator.
3. CR-03 (wifi), CR-07 (in-app bypass guard), CR-06 (`from==to`), CR-08 (content-change app re-check).
4. CR-04 (BypassCountStore cache), CR-05 (close() race).

**Track 3 — Architecture / debt (reduces future regression risk):**
1. ARCH-03 — `FileBackedStore<T>` shared abstraction (fixes CR-04 root, SEC-09).
2. ARCH-04 — shared schema-constants + CI schema check.
3. ARCH-05 (decompose god objects, after the evaluator), ARCH-06 (`BlockingContract`), ARCH-08 (delete dead `Settings` table), ARCH-09 (split channels), ARCH-07 (domain leaks), CR-10 (decision-function tests).
